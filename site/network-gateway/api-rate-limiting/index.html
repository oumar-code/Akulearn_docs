    Network Gateway API — Rate Limiting & Quotas¶

    Rate Limiting Strategy¶

    Tier-based limits¶

    Consumer Tier Requests/minute Burst Allowance Notes
    Admin Portal 1000 1200 Internal system, high trust
    Field Agents 300 350 Mobile app, moderate volume
    External Integrations 100 120 Third-party systems
    Webhooks (inbound) 50 60 Callback verification endpoints

    Rate limit headers¶

    All API responses include rate limit information:

    X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 950
X-RateLimit-Reset: 1737561600
    • X-RateLimit-Limit: Maximum requests allowed in current window
    • X-RateLimit-Remaining: Remaining requests before limit hit
    • X-RateLimit-Reset: Unix timestamp when limit resets

    429 Response format¶

    {
  "code": "rate_limit_exceeded",
  "message": "Rate limit of 1000 requests/minute exceeded",
  "detail": {
    "limit": 1000,
    "remaining": 0,
    "resetAt": "2026-01-22T11:00:00Z",
    "retryAfter": 45
  }
}

    Rate limit bypass (emergency)¶

    • X-RateLimit-Bypass: Include header with signed token for emergency operations
    • Requires platform admin approval
    • Logged and audited for compliance

    Batch Operations¶

    Batch size limits¶

    Operation Max Batch Size Processing Time (est)
    eSIM profile creation 100 30-60s
    pSIM ICCID registration 100 30-60s
    Policy updates 500 60-120s
    Profile suspension/resume 200 20-40s

    Batch workflow¶

    sequenceDiagram
    participant Client
    participant API as Gateway API
    participant Queue as Job Queue
    participant Worker as Batch Worker
    participant DB as Database

    Client->>API: POST /esim/profiles/batch
    API->>Queue: Enqueue batch job
    API->>Client: 202 Accepted (batchId)
    Queue->>Worker: Assign job
    loop Process each item
        Worker->>DB: Create profile
        Worker->>Queue: Update progress
    end
    Worker->>Queue: Mark complete
    Client->>API: GET /batch/{batchId}
    API->>Client: 200 OK (results)

    Batch status polling¶

    • Poll /batch/{batchId} for status updates
    • Recommended interval: 5 seconds
    • Status transitions: queued → processing → completed/failed
    • Results available for 24 hours after completion

    Partial failure handling¶

    • Batch continues even if individual items fail
    • Response includes per-item status and error details
    • Retry failed items with new batch request or individual API calls

    Webhook Integration¶

    Webhook registration¶

    POST /webhooks
{
  "url": "https://example.com/webhooks/sim-events",
  "events": ["profile.activated", "iccid.registered", "attach.success"],
  "secret": "wh_secret_abc123def456"
}

    Supported events¶

    Event Trigger Payload Data
    profile.created eSIM profile created profileId, deviceId, policyId
    profile.activated Device first attach profileId, imsi, cellId
    profile.suspended Admin/system suspend profileId, reason
    profile.deleted Profile removed profileId, deletedBy
    iccid.registered pSIM registered iccid, subscriberId, agent
    iccid.activated Device first attach iccid, imsi, deviceType
    iccid.suspended Admin/system suspend iccid, reason
    attach.success Successful attach imsi, cellId, rat
    policy.pushed Policy update delivered profileId/iccid, policyVersion
    swap.completed SIM swap finished oldId, newId, subscriberId

    Webhook payload format¶

    {
  "event": "profile.activated",
  "timestamp": "2026-01-22T10:20:45Z",
  "data": {
    "profileId": "profile-123",
    "deviceId": "device-456",
    "imsi": "310260123456789",
    "cellId": "edge-hub-01"
  },
  "signature": "sha256=a1b2c3d4e5f6789012345678901234567890abcdef..."
}

    Signature verification¶

    HMAC-SHA256 signature:

    import hmac
import hashlib

def verify_webhook(payload, signature, secret):
    expected = hmac.new(
        secret.encode(),
        payload.encode(),
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(f"sha256={expected}", signature)

    Header:

    X-Webhook-Signature: sha256=a1b2c3d4e5f6...

    Delivery guarantees¶

    • Retry policy: Exponential backoff (5s, 15s, 45s, 135s, 405s)
    • Max retries: 5 attempts
    • Timeout: 10 seconds per attempt
    • Success criteria: HTTP 200-299 response
    • Failure handling: Mark webhook inactive after 20 consecutive failures

    Circuit Breaker Pattern¶

    Gateway-level protection¶

    stateDiagram-v2
    [*] --> Closed: Normal operation
    Closed --> Open: Failure threshold exceeded
    Open --> HalfOpen: Timeout elapsed
    HalfOpen --> Closed: Test requests succeed
    HalfOpen --> Open: Test requests fail

    Configuration¶

    • Failure threshold: 50% error rate over 60s window
    • Open duration: 30 seconds
    • Half-open test: 3 requests
    • Recovery: Return to closed after 10 consecutive successes

    Error responses during circuit open¶

    {
  "code": "circuit_breaker_open",
  "message": "Gateway temporarily unavailable due to high error rate",
  "detail": {
    "service": "hlr_gateway",
    "reopenAt": "2026-01-22T10:30:00Z",
    "retryAfter": 30
  }
}

    Integration Testing¶

    Test environments¶

    Environment Base URL Rate Limits Purpose
    Sandbox https://sandbox-api.akulearn.com/network-gateway 10x production Integration testing
    Staging https://staging-api.akulearn.com/network-gateway 1x production Pre-production validation
    Production https://api.akulearn.com/network-gateway Full limits Live operations

    Test data¶

    • Sandbox: Auto-resets daily; pre-loaded test devices and policies
    • Test ICCIDs: Range 890123400000000000-890123400009999999 reserved for testing
    • Test profiles: Auto-expire after 24 hours in sandbox

    Peak load 500 5 min <2% errors, P95 <1s
    Burst 1000 30s <5% errors, P95 <2s
    Batch operations 50 batches/min 10 min <1% batch failures

    Quota increase requests¶

    • Justification required (use case, volume projection)
    • Approval within 2 business days
    • Temporary increases available for pilot/testing
