    Aku Network Gateway — eSIM¶

    Overview¶

    Aku eSIM provides remote provisioning and lifecycle management for embedded SIMs used by Aku Edge Hubs and connected devices. It integrates with the network gateway identity system and supports secure OTA provisioning, network selection, and provisioning policies.

    Integration with Aku Platform¶

    • Identity: integrates with the IG-Hub IdP for device identity and attestation.
    • Connectivity: interacts with Aku eSIM management APIs to provision profiles and manage network switching.
    • Aku Workspace: workspace workflows may surface eSIM provisioning status or require connectivity checks before performing remote tasks; see ../services/aku-workspace.md for cross-linking.

    Provisioning flow¶

    • Profile creation via IG-Hub with policy attached (local breakout preference, fallback MNOs).
    • Activation delivered by QR code or push for compatible devices.
    • Post-activation: device registers with Aku network gateway; applies network selection policy.
    sequenceDiagram
    participant Admin
    participant IG as IG-Hub
    participant NG as Network Gateway
    participant Dev as Device (eSIM)
    Admin->>IG: Create eSIM profile + policy
    IG->>NG: Push profile metadata
    Admin->>Dev: Deliver QR / activation code
    Dev->>NG: Download & install profile
    Dev->>NG: Attach + register
    NG->>Dev: Policy (breakout pref, retry budget)
    NG->>IG: Activation status + KPIs

    Provisioning error handling¶

    Profile creation failures:

    flowchart TD
    START[Create eSIM Request] --> VALIDATE{Validation}
    VALIDATE -->|Pass| CHECK_DEVICE{Device compatible?}
    VALIDATE -->|Fail| ERR1[400: Invalid input]
    CHECK_DEVICE -->|Yes| CHECK_QUOTA{Profile quota?}
    CHECK_DEVICE -->|No| ERR2[400: Incompatible device]
    CHECK_QUOTA -->|Available| CREATE[Create profile]
    CHECK_QUOTA -->|Exceeded| ERR3[409: Quota exceeded]
    CREATE -->|Success| PUSH[Push to gateway]
    CREATE -->|Fail| ERR4[500: Profile creation failed]
    PUSH -->|Success| NOTIFY[Notify admin + send QR]
    PUSH -->|Fail| RETRY{Retry count?}
    RETRY -->|< 3| PUSH
    RETRY -->|>= 3| ERR5[503: Gateway unavailable]

    Activation failures: - QR scan fails: Regenerate activation code; verify device camera/network - Profile download timeout: Check device connectivity; retry with exponential backoff (5s, 15s, 45s) - Install fails: Verify eSIM slot availability; check for conflicting profiles; log device model for compatibility review - Registration timeout: Verify gateway HLR/HSS reachability; check IMSI provisioning; escalate if persistent

    Profile lifecycle state machine¶

    stateDiagram-v2
    [*] --> Created: Profile provisioned
    Created --> Active: Device activates
    Created --> Expired: 7 days no activation
    Active --> Suspended: Admin/system suspend
    Active --> Deleted: Admin delete
    Suspended --> Active: Admin resume
    Suspended --> Deleted: Admin delete after 30d
    Expired --> Deleted: Auto-cleanup
    Deleted --> [*]

    Active --> Swapping: SIM swap initiated
    Swapping --> Deleted: Old profile invalidated

    State transition rules: - Created → Active: Device completes first attach; emits profile.activated event - Active → Suspended: Manual suspend or payment failure; device denied attach; emits profile.suspended - Suspended → Active: Manual resume or payment received; device allowed attach; emits profile.resumed - Active → Swapping: SIM swap request; old profile marked for deletion; emits profile.swap_initiated - Created → Expired: 7 days no activation; auto-expire and cleanup; emits profile.expired

    Known limitations¶

    • Dual SIM devices: Some models limit eSIM to secondary slot; verify slot priority config
    • Carrier locks: Devices locked to other carriers may reject eSIM profiles; require unlock first
    • OS updates: Major OS updates may require eSIM profile refresh; document user impact
    • Regional variants: Some models ship without eSIM in certain markets; verify IMEI prefix

    Expansion roadmap¶

    • Q2 2026: Add Xiaomi, Realme mid-range devices
    • Q3 2026: Test tablets (iPad, Galaxy Tab) for classroom deployments
    • Q4 2026: Validate Windows laptops with eSIM (Surface, ThinkPad)

    Security model¶

    • Mutual TLS for API calls to the eSIM management service.
    • Device attestation at first registration; periodic token renewal bound to device identity.
    • Audit logs for profile lifecycle (create, enable, suspend, revoke).

    Network switching policy¶

    • Shared policy and flow diagram: see network-gateway/index.md.
    • Prefer Aku edge/super-hub coverage; degrade to partner MNO when unavailable.
    • Policy updates delivered OTA; devices must honor max-retry/backoff to avoid flapping.

    A->>EDGE: Terminate call

    sequenceDiagram
    participant A as Device A (eSIM)
    participant GW as Gateway

    A->>GW: Initiate call to external
    GW->>MNO: No edge coverage, route via MNO
    MNO->>EXT: Interconnect

    Note over A,EXT: Voice path via MNO
    A->>MNO: Terminate

    sequenceDiagram
    participant DEV as Device (eSIM)
    participant GW as Gateway

    GW->>EDGE: Policy: prefer local breakout

    QoS Rules¶

    Tier definitions¶

    Tier Use Case Max Throughput Priority Throttle After
    Zero-rated Education content (allow-listed domains) 5 Mbps High Never
    Standard General browsing, social media 3 Mbps Medium 5 GB/month
    Premium HD video, large downloads 10 Mbps High 20 GB/month
    Voice VoIP, SIP calls 128 Kbps (G.711) Highest Never

    QoS enforcement¶

    • Zero-rated content: Bypass metering; always high priority; domain/IP allow-list maintained by policy engine.
    • Fair-use throttling: After bundle exhausted, reduce to 512 Kbps; notify via USSD.
    • Voice priority: SIP/RTP packets marked DSCP EF; guaranteed 128 Kbps minimum.
    • Congestion: Apply weighted fair queuing; voice > zero-rated > premium > standard.

    Policy updates¶

    • OTA push when user upgrades bundle or policy changes.
    • Device applies new QoS immediately; no reattach required.
    • Fallback to cached policy if push fails; retry on next attach.

    Dashboards¶

    KPI Dashboard (Grafana): - Activation success rate (target: >98%) - Attach success rate (target: >99%) - Policy push success rate (target: >99%) - Breakout share: Aku vs MNO (target: >70% Aku) - Active profiles gauge - P95 activation latency (target: <120s)

    Operational Dashboard: - Attach attempts by cell (edge-hub-01, edge-hub-02, etc.) - Profile state distribution (active, suspended, created, swapping) - Error rate by type (auth_failure, device_unreachable, quota_exceeded) - Data usage by tier (zero-rated, standard, premium)

    Alerting rules: - Activation success rate <95% for 10 min → Page on-call - Attach failure rate >2% for 5 min → Alert ops - Policy push failure rate >5% for 10 min → Alert engineering - Gateway API latency P95 >2s for 5 min → Alert platform team
