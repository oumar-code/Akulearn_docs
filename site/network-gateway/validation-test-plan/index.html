    Network Gateway Validation & Test Plan¶

    Overview¶

    Comprehensive test plan for validating eSIM and physical SIM provisioning, policy enforcement, and network gateway operations before pilot deployment.

    Test Phases: 1. Lab Validation — Controlled environment with instrumented devices 2. Integration Testing — Cross-system validation with billing, CRM, analytics 3. Field Pilot — Real-world validation with limited user cohort 4. Performance Testing — Load, stress, and scalability validation

    1. Lab Validation Scenarios¶

    1.1 eSIM Provisioning Tests¶

    Test Case: E-001 — QR Code Activation (Smart Device)¶

    Objective: Validate end-to-end eSIM profile download and activation via QR code

    Pre-conditions: - Test device: iPhone 12 or Samsung Galaxy S21 - Device in range of test cell tower - Backend API accessible

    Test Steps: 1. Generate eSIM profile via API: POST /esim/profiles 2. Display QR code containing SM-DP+ address and activation code 3. Scan QR code with test device 4. Monitor profile download progress 5. Activate profile in device settings

    Expected Results: - ✅ Profile downloads in <30 seconds - ✅ ICCID registered in HLR/HSS - ✅ Device attaches to Aku network within 60 seconds - ✅ profile.activated event fired to webhook subscribers - ✅ Default policy (zero-rated tier) applied

    Acceptance Criteria: - Success rate: ≥95% across 20 test runs - No error codes 4001-4005 encountered - Network registration time: P95 <90 seconds

    Test Case: E-002 — Manual Activation Code Entry¶

    Objective: Validate eSIM activation via manual entry (no QR scan)

    Test Steps: 1. Generate profile and retrieve activation code 2. Navigate to device eSIM settings 3. Select "Enter Details Manually" 4. Input SM-DP+ address and activation code 5. Complete activation

    Expected Results: - ✅ Manual entry successful on iOS and Android - ✅ Profile activation completes in <45 seconds - ✅ Same backend validation as QR code flow

    Acceptance Criteria: - Success rate: ≥90% (accounts for user input errors) - Clear error messages for invalid codes

    Test Case: E-003 — Profile Swap (eSIM to eSIM)¶

    Objective: Validate swapping active profile between two eSIM-capable devices

    Test Steps: 1. Activate profile on Device A 2. Call POST /esim/profiles/{id}/swap with Device B IMEI 3. Deactivate profile on Device A 4. Activate profile on Device B 5. Verify network access on Device B

    Expected Results: - ✅ Swap completes in <90 seconds - ✅ Device A loses network access - ✅ Device B gains network access - ✅ swap.completed event with both IMEIs - ✅ No data loss or billing interruption

    Acceptance Criteria: - Swap success rate: ≥98% - Downtime: <30 seconds between deactivation and reactivation - No orphaned profiles in HLR

    1.2 Physical SIM Tests¶

    Test Case: P-001 — ICCID Registration (Feature Phone)¶

    Objective: Validate physical SIM registration and activation on feature phone

    Pre-conditions: - Test device: Nokia 105 or similar feature phone - Pre-loaded SIM card with blank profile

    Test Steps: 1. Insert SIM into device 2. Field agent registers ICCID via API: POST /psim/iccids 3. Assign MSISDN to ICCID 4. Trigger HLR provisioning 5. Power cycle device

    Expected Results: - ✅ ICCID registered in backend within 5 seconds - ✅ HLR entry created with MSISDN - ✅ Device attaches to network after reboot - ✅ iccid.activated event fired - ✅ USSD menu accessible via *123#

    Acceptance Criteria: - Registration success rate: ≥99% (robust ICCID validation) - Network attach time: P95 <120 seconds - USSD response time: <5 seconds

    Test Case: P-002 — SIM Swap (Physical to Physical)¶

    Objective: Validate swapping physical SIM between devices

    Test Steps: 1. Activate SIM in Device A (feature phone) 2. Call POST /psim/iccids/{iccid}/swap with Device B IMEI 3. Move SIM card to Device B 4. Power on Device B 5. Verify network access and MSISDN

    Expected Results: - ✅ Backend swap recorded immediately - ✅ HLR updated with new IMEI - ✅ Device B attaches to network - ✅ SMS and voice services functional - ✅ swap.completed event fired

    Acceptance Criteria: - Swap success rate: ≥99% - No service interruption if SIM moved during active call (graceful disconnect) - IMEI update propagates to all systems within 10 seconds

    Test Case: P-003 — Lost/Stolen SIM Suspension¶

    Objective: Validate SIM suspension and unsuspension workflow

    Test Steps: 1. Activate SIM in test device 2. Mark as lost: PATCH /psim/iccids/{iccid} with status: LOST 3. Verify network access denied 4. Mark as found: PATCH /psim/iccids/{iccid} with status: ACTIVE 5. Verify network access restored

    Expected Results: - ✅ Suspension takes effect in <10 seconds - ✅ Device loses network registration - ✅ Unsuspension restores access in <20 seconds - ✅ iccid.suspended and iccid.activated events fired - ✅ No billing during suspended period

    Acceptance Criteria: - Suspension success rate: 100% (critical security feature) - Event propagation to billing: <30 seconds

    1.3 Policy Enforcement Tests¶

    Test Case: PL-001 — Zero-Rated Content Access¶

    Objective: Validate zero-rated policy allows Akulearn content without data deduction

    Test Steps: 1. Activate SIM with zero-rated tier 2. Access Akulearn API endpoints (lessons, quizzes, videos) 3. Monitor data usage via GET /data-usage/{iccid} 4. Access non-Akulearn site (e.g., google.com) 5. Verify data deduction for non-zero-rated traffic

    Expected Results: - ✅ Akulearn traffic not metered - ✅ Non-Akulearn traffic deducted from data bundle - ✅ Policy enforcement at PGW/PCEF layer - ✅ Billing records show zero-rated vs. standard traffic split

    Acceptance Criteria: - Zero-rating accuracy: 100% (no false billing) - Policy push time: <5 seconds after profile activation

    Test Case: PL-002 — QoS Tier Transition¶

    Objective: Validate seamless transition between QoS tiers

    Test Steps: 1. Start with standard tier (2 Mbps) 2. Upgrade to premium tier (10 Mbps): POST /policies/push 3. Run speed test before and after 4. Downgrade to standard tier 5. Run speed test again

    Expected Results: - ✅ Speed test reflects tier limits (±10% tolerance) - ✅ Tier change takes effect in <10 seconds - ✅ No connection drop during transition - ✅ policy.pushed event with new tier

    Acceptance Criteria: - Tier transition success rate: ≥98% - No user-visible disruption (TCP connections maintained)

    Test Case: PL-003 — Voice QoS Priority¶

    Objective: Validate voice traffic prioritized over data

    Test Steps: 1. Activate SIM with voice tier 2. Start large file download (saturate link) 3. Place voice call during download 4. Monitor voice quality (MOS score) and latency

    Expected Results: - ✅ Voice call completes without dropouts - ✅ MOS score ≥4.0 (toll quality) - ✅ Voice latency <150ms - ✅ Data throughput throttled during call

    Acceptance Criteria: - Voice quality degradation: <5% compared to idle network - Call setup time: P95 <3 seconds

    1.4 Network Selection Tests¶

    Test Case: N-001 — Aku Edge Network Preference¶

    Objective: Validate device prefers Aku edge network when in coverage

    Test Steps: 1. Place test device in area with Aku edge coverage 2. Power on device with active SIM 3. Monitor network registration via GET /network-status/{iccid} 4. Verify PLMN ID matches Aku edge

    Expected Results: - ✅ Device registers to Aku edge PLMN - ✅ Data routed through Aku PGW - ✅ Zero-rated policy applied at Aku PCEF - ✅ attach.success event with network: aku-edge

    Acceptance Criteria: - Aku edge selection rate: ≥95% when in coverage - Attach time: P95 <60 seconds

    Test Case: N-002 — MNO Fallback¶

    Objective: Validate fallback to partner MNO when outside Aku coverage

    Test Steps: 1. Start with device on Aku edge network 2. Move device to area without Aku coverage 3. Monitor handover to partner MNO 4. Verify data access continues

    Expected Results: - ✅ Device reselects to partner MNO PLMN - ✅ Data routed through MNO PGW - ✅ Policy enforced via MNO PCEF (roaming agreement) - ✅ attach.success event with network: mno-fallback - ✅ User experience seamless (no reconnection required)

    Acceptance Criteria: - Handover success rate: ≥90% - Handover time: P95 <30 seconds - No data loss during handover

    Test Case: N-003 — Network Selection Override¶

    Objective: Validate manual network selection for testing/troubleshooting

    Test Steps: 1. Enable manual network selection in device settings 2. Select partner MNO PLMN manually 3. Verify registration to chosen network 4. Return to automatic selection 5. Verify device returns to Aku edge

    Expected Results: - ✅ Manual selection honored - ✅ Backend aware of network override: network_override: true - ✅ Policies applied via roaming interface - ✅ Automatic selection restores preferred behavior

    Acceptance Criteria: - Manual selection reliability: ≥95% - Override flag propagates to backend within 60 seconds

    2. Integration Testing¶

    2.1 Billing System Integration¶

    Test Case: B-001 — Activation Event to Billing¶

    Objective: Validate profile/ICCID activation events trigger billing records

    Test Steps: 1. Activate eSIM profile 2. Monitor webhook delivery to billing system 3. Verify billing record created with: - ICCID/profile ID - Activation timestamp - Initial plan (free tier) - MSISDN

    Expected Results: - ✅ profile.activated event delivered to billing webhook - ✅ Billing record created within 5 seconds - ✅ HMAC signature valid - ✅ Idempotent (duplicate events ignored)

    Acceptance Criteria: - Event delivery success rate: ≥99.9% - Billing record accuracy: 100%

    Test Case: B-002 — Data Usage Synchronization¶

    Objective: Validate real-time data usage reporting to billing

    Test Steps: 1. Activate SIM and consume 10 MB of data 2. Query backend: GET /data-usage/{iccid} 3. Query billing system for same ICCID 4. Compare usage values

    Expected Results: - ✅ Usage values match (±1% tolerance) - ✅ Usage updated in billing within 60 seconds - ✅ Zero-rated and standard traffic tracked separately

    Acceptance Criteria: - Usage accuracy: ≥99% - Sync latency: P95 <90 seconds

    2.2 CRM Integration¶

    Test Case: C-001 — Customer Profile Linking¶

    Objective: Validate ICCID/profile linked to customer record in CRM

    Test Steps: 1. Create customer in CRM with user ID 2. Provision SIM/profile linked to user ID 3. Query CRM for customer SIM inventory 4. Verify ICCID/profile appears in inventory

    Expected Results: - ✅ CRM inventory updated via webhook - ✅ Link established between customer and SIM - ✅ Swap events update CRM device records

    Acceptance Criteria: - Link success rate: ≥99% - CRM sync latency: P95 <120 seconds

    2.3 Analytics Platform Integration¶

    Test Case: A-001 — Usage Pattern Analysis¶

    Objective: Validate network events stream to analytics for insights

    Test Steps: 1. Generate 100 attach events across 10 devices 2. Query analytics platform for attach rate metrics 3. Verify event volume and distribution

    Expected Results: - ✅ Events delivered to analytics webhook - ✅ Event schema validated (no malformed payloads) - ✅ Metrics dashboard reflects activity within 5 minutes

    Acceptance Criteria: - Event delivery rate: ≥99% - Analytics latency: P95 <5 minutes (batch processing acceptable)

    3. Field Pilot Test Cases¶

    3.1 Pilot Cohort Definition¶

    Pilot Size: 100 users across 3 regions - Region A: Urban area with Aku edge coverage (40 users) - Region B: Semi-urban area with partial coverage (40 users) - Region C: Rural area with MNO-only coverage (20 users)

    Device Distribution: - eSIM-capable smartphones: 60 users (iPhone, Samsung, Google Pixel) - Feature phones: 40 users (Nokia, Itel, Tecno)

    Pilot Duration: 4 weeks

    3.2 Pilot Test Scenarios¶

    Scenario: FP-001 — Field Agent Onboarding¶

    Objective: Validate field agent can provision SIMs without technical support

    Steps: 1. Train field agent on runbook (1-hour session) 2. Provide test device and SIM inventory 3. Agent registers 10 SIMs using mobile app 4. Agent activates SIMs and verifies network access

    Success Metrics: - Agent completes training and passes quiz (≥80% score) - Registers 10 SIMs in <30 minutes - Zero escalations to technical support

    Scenario: FP-002 — User Activation Experience¶

    Objective: Validate end-user can activate SIM with minimal support

    Steps: 1. Distribute SIM cards to 10 pilot users 2. Provide QR code / activation instructions 3. Users self-activate (no agent assistance) 4. Survey users on experience (1-5 scale)

    Success Metrics: - Self-activation success rate: ≥80% - User satisfaction score: ≥4.0/5.0 - Support contact rate: <10%

    Scenario: FP-003 — Daily Usage Patterns¶

    Objective: Capture real-world usage to tune policies and capacity

    Monitoring (4 weeks): - Data consumption per user per day - Peak usage hours - Network selection (Aku vs. MNO) - Content type (zero-rated vs. standard) - Device handover frequency - Support ticket volume

    Target Metrics: - Average data usage: 200-500 MB/user/day - Peak hours: 6-9 PM (expect 3x baseline traffic) - Aku edge usage: ≥70% in Region A - Support tickets: <5 per 100 users per week

    Scenario: FP-004 — Support Escalation Testing¶

    Objective: Validate support team can resolve common issues using runbook

    Test Cases: 1. User reports "No network access" (SIM not activated) 2. User reports "Can't download profile" (QR code error) 3. User reports "Lost SIM" (requires suspension) 4. User needs device swap (same ICCID, new IMEI)

    Success Metrics: - Support resolves 80% of issues on first contact - Average resolution time: <10 minutes - Escalation rate to engineering: <5%

    4. Performance & Load Testing¶

    4.1 API Load Tests¶

    Load Test: L-001 — Concurrent Registrations¶

    Objective: Validate backend handles 100 concurrent SIM registrations

    Configuration: - 100 virtual users - Each registers 1 SIM via POST /esim/profiles or POST /psim/iccids - Ramp-up: 10 users/second - Sustained load: 60 seconds

    Expected Results: - ✅ All 100 registrations succeed - ✅ P95 latency: <2 seconds - ✅ No 500 errors - ✅ Database connections stable (<80% pool utilization)

    Acceptance Criteria: - Success rate: ≥99% - Error rate: <1%

    Load Test: L-002 — Policy Push Burst¶

    Objective: Validate policy engine handles burst of 500 policy updates

    Configuration: - 500 policy updates via POST /policies/push - No ramp-up (instant burst) - Target: 500 updates in <10 seconds

    Expected Results: - ✅ All policies pushed successfully - ✅ P95 latency: <3 seconds - ✅ Circuit breaker does not trip - ✅ Message queue depth returns to zero within 30 seconds

    Acceptance Criteria: - Success rate: ≥98% - No policy conflicts or race conditions

    4.2 Network Load Tests¶

    Load Test: N-001 — Simultaneous Attach Events¶

    Objective: Validate HLR/HSS handles 100 simultaneous device attaches

    Configuration: - 100 test devices powered on simultaneously - Monitor attach success rate and time

    Expected Results: - ✅ ≥95 devices attach successfully - ✅ P95 attach time: <90 seconds - ✅ No HLR/HSS crashes or timeouts

    Acceptance Criteria: - Attach success rate: ≥95% - System stability maintained (no restarts required)

    5. Acceptance Criteria Summary¶

    Critical (Must Pass Before Pilot)¶

    • [ ] eSIM activation success rate: ≥95%
    • [ ] Physical SIM activation success rate: ≥99%
    • [ ] Policy enforcement accuracy: 100% (no billing errors)
    • [ ] Swap operation success rate: ≥98%
    • [ ] Webhook delivery success rate: ≥99.9%
    • [ ] Support runbook resolves 80% of issues on first contact

    Important (Must Pass Before General Availability)¶

    • [ ] Network handover success rate: ≥90%
    • [ ] API P95 latency: <2 seconds
    • [ ] Load test success rate: ≥99%
    • [ ] User satisfaction score: ≥4.0/5.0
    • [ ] Field agent training completion rate: 100%

    Nice-to-Have (Iterative Improvement)¶

    • [ ] Analytics dashboard real-time updates (<1 minute latency)
    • [ ] USSD menu response time: <3 seconds
    • [ ] Manual network selection reliability: ≥95%

    7. Test Execution Schedule¶

    Week 1-2: Lab Validation¶

    • Days 1-3: eSIM provisioning tests (E-001 to E-003)
    • Days 4-6: Physical SIM tests (P-001 to P-003)
    • Days 7-10: Policy enforcement tests (PL-001 to PL-003)
    • Days 11-14: Network selection tests (N-001 to N-003)

    Week 3: Integration Testing¶

    • Days 15-17: Billing integration (B-001 to B-002)
    • Days 18-19: CRM integration (C-001)
    • Days 20-21: Analytics integration (A-001)

    Week 4: Performance Testing¶

    • Days 22-24: API load tests (L-001 to L-002)
    • Days 25-26: Network load tests (N-001)
    • Day 27: Results analysis and tuning

    Week 5-8: Field Pilot¶

    • Week 5: Onboarding and initial activation (FP-001, FP-002)
    • Week 6-7: Daily usage monitoring (FP-003)
    • Week 8: Support escalation testing (FP-004) and feedback collection

    8. Test Reporting¶

    Daily Standups¶

    Attendees: QA lead, backend engineer, network engineer, pilot coordinator

    Agenda: - Test cases executed (pass/fail count) - Blockers and issues - Adjustments to test plan

    Weekly Reports¶

    Metrics: - Test case coverage: X/Y completed - Defect count by severity (Critical, High, Medium, Low) - Performance benchmarks vs. targets - Risk assessment updates

    Pilot Weekly Reports¶

    Metrics: - Active users: X/100 - Activation success rate: Y% - Support ticket volume and categories - User feedback summary (satisfaction scores, feature requests) - Network performance (attach success, handover success, Aku vs. MNO usage)

    9. Go/No-Go Decision Criteria¶

    Go Criteria (Proceed to General Availability)¶

    • ✅ All critical acceptance criteria met
    • ✅ Zero critical defects open
    • ✅ High-severity defects: <3 open
    • ✅ Pilot user satisfaction: ≥4.0/5.0
    • ✅ Support escalation rate: <5%
    • ✅ Performance targets met under load

    No-Go Criteria (Extend Pilot or Return to Development)¶

    • ❌ Any critical acceptance criteria failed
    • ❌ Critical defects open
    • ❌ Pilot user satisfaction: <3.5/5.0
    • ❌ Support escalation rate: >10%
    • ❌ Performance degradation under load
