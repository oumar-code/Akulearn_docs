# Hackathon Challenge Execution Strategy

> **Status:** Adopted  
> **Audience:** Product, AI/ML, Engineering, Demo/Submission Team  
> **Last updated:** July 2026  
> **Related docs:** [`strategy/aku-code-editor-strategy.md`](./aku-code-editor-strategy.md) · [`strategy/rag_chatbot_strategy.md`](./rag_chatbot_strategy.md)

---

## Executive Direction

Run one core product concept across all target challenges and adapt positioning, optimization focus, and submission narrative per organizer criteria.

Core principle: **single technical backbone, challenge-specific value story.**

---

## 1) Core Product Backbone (Build Once, Adapt Fast)

Define one product baseline with shared architecture:

- Ingestion + orchestration layer
- Model serving layer
- Evaluation + benchmark layer
- Demo UI layer
- Submission asset package

Per challenge, only adjust:

- Use-case framing
- Model/profile tuning
- Infrastructure configuration
- KPI emphasis in the pitch

---

## 2) Three Parallel Tracks

### Build Track

- Deliver a reliable prototype early
- Stabilize demo flow and fallback states
- Keep deployment reproducible

### AI Track

- Tune model quality and latency
- Run controlled evaluations
- Optimize for target infrastructure limits

### Submission Track

- Build storyline and judge-facing narrative
- Prepare deck, technical write-up, and demo video
- Keep final artifacts submission-ready from mid-cycle

---

## 3) Challenge-by-Challenge Alignment

### Arm Create: AI Optimization Challenge

Prioritize:

- Inference speed
- Memory footprint
- Cost per run
- Edge deployment readiness

Judge-facing proof:

- Baseline vs optimized benchmark table
- Hardware-aware optimization decisions
- Repeatable edge deployment evidence

### Backblaze Generative Media Hackathon (Genblaze on B2)

Prioritize:

- Generative media workflow quality
- End-to-end B2 storage pipeline
- Throughput and scaling reliability

Judge-facing proof:

- Media generation pipeline architecture
- B2 lifecycle, storage, and retrieval flow
- Performance and scalability measurements

### Global AI Hackathon Series (Qwen Cloud)

Prioritize:

- Strong multilingual behavior
- Prompt/fine-tuning quality controls
- Cloud-native deployment and operations

Judge-facing proof:

- Multilingual evaluation results
- Cloud deployment architecture and observability
- Practical production-readiness narrative

---

## 4) Four-Phase Event Cadence

### Day 1 — Scope and Architecture

- Lock problem statement
- Confirm architecture
- Define measurable success metrics

### Days 2–3 — MVP and Baseline Demo

- Ship end-to-end MVP
- Record baseline quality, latency, and cost
- Produce first demo cut

### Days 4–5 — Optimization and Polish

- Optimize bottlenecks
- Incorporate user/test feedback
- Finalize story and visual assets

### Final Day — Submission Packaging

- Execute final live demo rehearsal
- Export backup recorded demo
- Finalize pitch deck and technical report
- Submit complete artifact bundle

---

## 5) Judging-Ready Asset Kit (Prepared Early)

Maintain these assets before final day:

- 2-minute live demo script
- Backup full demo recording
- Architecture diagram
- Benchmark table (latency / cost / quality)
- Impact narrative (who benefits, why now, why this team)

---

## 6) Risk-Control Policy

- Freeze feature scope 24 hours before deadline
- Keep offline and prerecorded demo fallback
- Maintain two model paths:
  - **Safe path:** stable, validated configuration
  - **Aggressive path:** experimental, higher upside optimization

Escalate to safe path if aggressive path threatens submission reliability.

---

## 7) Winning Focus

To maximize scoring:

- Solve one painful problem deeply
- Show measurable, verifiable improvement
- Demonstrate clear post-hackathon shipping path

Avoid broad but shallow feature sets; prioritize depth, credibility, and demonstrable impact.
